# central-sports-web 仕様書

## 1. 導入

### 1.1 目的

セントラルスポーツの予約システムに対し、事前登録した予約ターゲットを毎朝 9 時ちょうどに自動で予約する常駐 Web アプリ。人気プログラムが数秒で埋まる状況で、人手では取りきれない枠を確実に確保する。

### 1.2 スコープ

**対象**:

- 予約ターゲットの事前登録と自動予約（9 時発火）。
- 予約の即時作成・キャンセル・席変更をブラウザから操作。
- 月間スケジュールと直近 1 週間の実スケジュールを重ねたカレンダー表示。
- 複数店舗の切替（初期デフォルトは府中 第1スタジオ）。

**Non-Goals**:

- 施設入館の自動チェックイン（到着検知の複雑さに対し効果が限定的なため本フェーズ外）。
- 多アカウント対応（将来拡張、本フェーズでは 1 アカウント前提）。
- レッスン料金の計算・決済（既存会員契約でカバー）。
- 予約以外の会員機能（情報変更、明細閲覧等）。

### 1.3 ステークホルダー

個人会員 1 名による自家利用。平日は本業で日中に画面操作する余裕がなく、朝の予約枠開放に手動で参加できない。夜間や休日に事前登録を行い、当日朝の結果を通知で受け取りたい。技術的には Web アプリの運用経験がある利用者。

### 1.4 設計方針

- **常駐プロセス化**: 9 時発火の応答速度を確保するため、事前ログイン済みのセッションとスケジューラをプロセス内に持つ。毎回コンテナを起動する方式はコンテナ起動に 14 秒を要し、数秒の競争で敗北するため不採用。
- **利用規約準拠の通信互換性**: ブラウザと区別されにくい HTTP クライアント（市場-platform の既存方針を踏襲）を使い、通常利用と区別されない範囲のヘッダセットで通信する。過剰な polling は行わず、9 時発火時のみ正当な予約行動として POST する。
- **既存 CLI との共用**: 事前調査用に作成した API クライアント層と復号ユーティリティを Web アプリ側から再利用する。CLI は保守用に残置し、同じコードパスを通すことで動作の一貫性を担保する。
- **状態の外部化**: ターゲット定義・予約結果・実行ログ・スケジュールキャッシュは永続ストアに寄せ、プロセス再起動に耐える。認証トークンのみメモリ保持とし、失効時に自動再ログインする。

## 2. システム概要

### 2.1 アーキテクチャ

<img src="diagrams/system-context.svg" alt="システムコンテキスト図" width="80%">

**図 2.1: システムコンテキスト図**

Synology NAS 上の Docker コンテナとして常駐する。主要要素:

- **Web アプリ本体**: 利用者のブラウザからの操作を受け、永続ストアに対してターゲットや予約を読み書きする。外部サービスとのやり取りはセッション管理を経由する。
- **セッション管理**: 認証トークン（Cookie）をメモリ上で保持し、失効を検知したら再ログインする。外部の予約 API と直接通信する唯一のモジュール。
- **スケジューラ**: プロセス内で時刻ベースに起動し、毎朝 9 時前後の予約発火と、月次・週次の公開スケジュール同期を担う。
- **永続ストア**: ターゲット定義・予約結果・実行ログ・スケジュールキャッシュを保持する。プロセス再起動に耐える。

詳細なコンポーネント設計と技術選定は architecture.md に記載する。

### 2.2 データフロー

<img src="diagrams/data-flow.svg" alt="データフロー図" width="70%">

**図 2.2: データフロー図**

1. 月初に公開月間 API から翌月分の全プログラム定義を取得し、永続ストアにキャッシュする。
2. 週次に差分を再取得し、代行や枠変更を反映する。
3. 画面アクセス時に直近 1 週間の実スケジュールを予約 API から取得し、キャッシュを重ねてカレンダー表示する。
4. 利用者は画面で自動予約ターゲットを登録する。希望席は複数候補を指定できる。
5. 毎朝 8:55 頃にスケジューラが事前ログインと対象レッスン解決を行い、Discord に準備完了を通知する。
6. 9:00 ちょうどにスケジューラが各ターゲットに対して予約 POST を希望席順に試行する。
7. 結果は永続ストアに記録し、Discord で利用者に通知する。利用者はカレンダー UI で結果を確認する。

### 2.3 外部インターフェース

| 接続先 | 方式 | 用途 |
|---|---|---|
| 予約 API（reserve.central.co.jp） | HTTPS + Cookie 認証 | ログイン・スケジュール取得・予約作成/キャンセル/席変更 |
| 公開月間 API（www.central.co.jp） | HTTPS + JSONP | 月間スケジュール（代行反映前のテンプレート） |
| Discord | Webhook | 発火結果・障害通知 |
| 利用者ブラウザ | HTTPS + Basic 認証 | 画面操作 |

## 3. 機能要件

### 3.1 認証とセッション管理

会員メールアドレスとパスワードで外部予約 API にログインし、取得したアクセストークンを Cookie でメモリ保持する。失効検知（特定エラーコードの受信）で自動再ログインするが、連続失敗時はロックアウトを誘発しないよう所定回数で停止し通知する。

パスワードは外部シークレットストアから復号して利用するのみで、永続ストアやログに一切保存しない。UI への応答にも含めない。

### 3.2 スケジュール閲覧

月間テンプレートと直近 1 週間の実スケジュールを重ねたカレンダーを表示する。セルを選択するとレッスン詳細と座席の空き状況、自動予約ターゲット登録、即時予約・キャンセル・席変更の操作が可能。

月間テンプレートは公開 API 由来の代行反映前データ、直近 1 週間は予約 API 由来の代行反映後データとして扱う。満席・自動予約登録済・通常枠を視覚的に区別する。

### 3.3 店舗選択

利用者は店舗プルダウンで対象スタジオとルームを切り替えられる。初期デフォルトは府中 第1スタジオ（識別子は実装で保持）。店舗切替は表示のみに影響し、自動予約ターゲットは店舗横断で管理される。

### 3.4 予約の即時操作

利用者はカレンダーのセル詳細や予約一覧から以下の操作を行える:

- **予約作成**: 座席番号を選んで即時予約。予約 API の仕様に従い、指定席が埋まっている場合はエラーを表示する。
- **キャンセル**: 既存予約を取り消す。API のキャンセル可能期限に従う。
- **席変更**: 既存予約の座席番号を別の空き番号に変更する。

### 3.5 自動予約ターゲット管理

利用者はカレンダー上で曜日・時刻・プログラム名パターンをキーにターゲットを登録する。希望席は第 1 候補から任意数の候補まで並べて登録でき、発火時は順に試行する。インストラクター名はキーに含めない（代行で変わるため）。

ターゲットは有効化・無効化・削除ができる。登録済みセルはカレンダー上で視覚的に区別される。

マッチングは曜日と時刻の一致、プログラム名の部分一致もしくは正規表現で行う。同一条件に複数枠が該当した場合はプログラム名の最長一致を優先する。該当なしの週は通知のみ行い、予約は試行しない。

### 3.6 9 時発火

<img src="diagrams/target-state.svg" alt="ターゲット状態遷移図" width="70%">

**図 3.1: ターゲット状態遷移図**

毎朝 8:55 頃にスケジューラが事前準備を行い、9:00 ちょうどに予約 POST を発火する。発火は直前に実スケジュールを取得して対象レッスン識別子を確定させてから、希望席の第 1 候補から順に試行する。成功したら停止、すべて失敗したら通知のみとする。

同一ターゲットの同日二重発火は冪等キー（ターゲット識別子と予定日の組）で排他する。プロセス再起動中に発火時刻をまたいだ場合、所定の猶予時間内であれば 1 回だけ再発火を許容する。

ターゲットの状態は登録時に active、発火予定時刻に scheduled、発火中に running、結果に応じて succeeded・failed・skipped に遷移する。翌週に向けて active に戻る。手動で disabled に切り替えると発火対象から外れる。

### 3.7 手動即時発火

検証やリカバリのために、利用者はターゲットを選んで手動即時発火できる。挙動は 9 時発火と同じだが、準備通知は省略する。

### 3.8 通知

結果は Discord Webhook に投稿する。レベルは「成功要約」「失敗即時」「警告」の 3 段階。通知失敗は予約失敗として扱わず、実行ログに記録のみする。

## 4. 非機能要件

### 4.1 セキュリティ

- **UI アクセス制御**: Basic 認証もしくは Cookie ベース認証を必須とする。パスワードを握るアプリに LAN 内からでも無認証でアクセスできる状態は許容しない。
- **シークレットの分離**: 外部シークレットストアを読み取り専用でマウントし、復号結果はメモリ上のみで保持する。永続ストア・UI 応答・ログに含めない。
- **通信互換性**: 予約 API への通信はブラウザと区別されにくいクライアントで行う。事前調査用 CLI と同じ方針。

### 4.2 信頼性

- **重複実行防止**: ターゲット識別子と予定日の組を冪等キーとし、同日二重発火を排他する。
- **エラー分類**: 予約 API の応答コードを認証失効・既予約/満席・入力不正・サーバエラーに分類し、それぞれで再試行・停止・通知の方針を定める。
- **セッション失効対応**: 失効検知時は 1 回だけ再ログインを試行。連続失敗でロックアウトを誘発しないよう所定回数で停止し通知する。
- **先回り禁止**: 9:00 より早く予約 POST を送らない。遅延は許容する。
- **状態復元**: プロセス再起動後、永続ストアから状態を復元して通常運用に戻れる。

### 4.3 パフォーマンス

- **予約 POST の応答性**: 事前調査の実測で予約 POST は 210 ms 前後、ログイン込みで 700 ms 前後。9:00 ちょうどからの遅延は ±1 秒以内を目標とする。
- **カレンダー表示**: 月間テンプレートはキャッシュから返却し、画面初期表示は外部 API 呼び出し 1 回以内で完結する。
- **polling 抑制**: 空き監視などの定期 polling は原則行わない。行う場合でも分単位を下限とする。

### 4.4 制約条件

- 対象予約 API は非公開の内部 API。仕様変更時は実装の追従が必要。
- スケジュール開放は毎朝 9:00 の JST 固定で、1 週間先までしか見えない（実測仕様）。
- 予約 API の座席レイアウト（35/20/25 名枠）によって指定可能な座席番号の範囲が異なる可能性があり、実装着手前の追試で確定させる。
- ログイン失敗時のロックアウト機構の有無は未確認のため、リトライは慎重に設計する。

## 5. 運用

### 5.1 デプロイ

Synology NAS 上のコンテナとして常駐運用する。`restart: always` で NAS 再起動時も自動起動する。タイムゾーンは `Asia/Tokyo` 固定。

シークレットは NAS 上の専用ボリュームを読み取り専用でマウントする。アプリはパスワードを復号して使うのみで書き換えない。永続ストアはアプリ専用のボリュームに置き、週次で中身を JSON でエクスポートしてバックアップする。

### 5.2 監視・アラート

- **外形監視**: 起動状態・直近ログイン状態・スケジューラ状態を返すエンドポイントを提供し、NAS 側の外形監視基盤と連携する。
- **通知**: 予約発火・障害・セッション失効を Discord Webhook で通知する。通知レベル別に Embed のスタイルを変える。
- **実行ログ**: 全外部 API 呼び出しについて所要時間とリクエスト識別子を構造化ログに残し、成功率と失敗コード分布を確認できるようにする。

### 5.3 スケジュール

| ジョブ | 実行頻度 | 目的 |
|---|---|---|
| 月間スケジュール全件取得 | 月初 | 翌月のテンプレートを永続ストアにキャッシュ |
| 月間スケジュール差分取得 | 週次 | 代行や枠変更の反映 |
| 事前ログイン・対象解決 | 平日/日次 08:55 | 9 時発火の準備 |
| 予約発火 | 平日/日次 09:00 | 登録済みターゲットの予約 POST |
| 永続ストアのバックアップ | 週次 | JSON エクスポート |

平日・祝日・休館日の扱いは利用者の事前登録に従う（休館日は登録側で除外）。

### 5.4 障害対応

| 事象 | 対応 |
|---|---|
| 予約 API のログイン失敗（認証情報変更等） | 所定回数で停止、Discord に通知、以降のジョブは warning で保留 |
| 予約 POST の満席・既予約 | 次の希望席を試行、全滅なら通知のみ |
| 予約 API のサーバエラー（5xx） | 短時間バックオフで 1〜2 回再試行、超過で通知 |
| スケジューラプロセスの停止 | `restart: always` で再起動、所定猶予内なら発火再試行 |
| Discord Webhook の失敗 | 実行ログに記録のみ、予約処理は継続 |
| 永続ストアの破損 | 週次バックアップから復旧、復旧手順は運用マニュアル参照 |

自動復旧できないケースは Discord 通知で利用者の手動対応を促す。

## 6. 検討した代替案

- **毎回コンテナ起動方式**: 既存 CLI の延長で `docker compose run --rm` 方式。実測でコンテナ起動に 14 秒かかり、9 時レースに敗北する可能性が高いため不採用。
- **ブラウザ自動操作（Playwright 等）**: hacomono の API 呼び出しと同等の効果だが起動コストが重く、メモリ消費も大きい。HTTP クライアントに TLS 指紋互換を持たせる方式で十分と判断。
- **cron + シェルスクリプト**: 事前ログインの状態保持ができず、毎回ログインが必要で時間が読めない。常駐プロセス方式を採用。
- **外部 SaaS スケジューラ（AWS EventBridge 等）**: セッション持ち込みが難しく、コストも増える。NAS 上の常駐プロセスで完結させる。
