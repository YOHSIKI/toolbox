# central-sports-web アーキテクチャ設計書

## 1. 設計方針

### 1.1 アーキテクチャパターン

単一プロセス内に UI サーバ・スケジューラ・セッションキーパー・永続ストアアクセスを持つレイヤード構成とする。外部の他プロセスとの協調は行わず、状態はプロセス内メモリと単一の永続ストアに集約する。

層の分離は次の 4 つ:

- **プレゼンテーション層**: HTTP ルーティングと HTML テンプレート生成。
- **アプリケーションサービス層**: ユースケース単位の処理（予約作成、ターゲット登録、発火制御）。
- **外部連携層**: 予約 API・公開月間 API・Discord などの外部通信。セッションキーパーが集中管理する。
- **永続化層**: 永続ストアへの読み書き。書き込みは単一のキューで直列化する。

上位層は下位層にのみ依存する（逆向き依存を禁止）。外部連携層と永続化層は並列で、アプリケーションサービス層から呼ばれる。

### 1.2 技術スタック

Web アプリは Python 製。非同期 I/O ループ上の HTTP サーバを使うが、外部 HTTP クライアントは同期ライブラリを採用する（TLS 指紋互換のため）。ブロッキング処理はバックグラウンドスレッドに逃がして非同期ループを妨げない。

UI テンプレートはサーバサイドレンダリング。JavaScript は対話的な挙動が必要な箇所のみに限定する（SPA 化はしない）。

永続ストアは組み込み型の軽量 RDB を採用する。単一プロセスかつ低同時接続数で、運用コストを抑えるため。

スケジューラはプロセス内常駐のものを採用し、発火ジョブと月次/週次同期ジョブを同一プロセスで扱う。

外部依存ライブラリのバージョンはリポジトリ内のロックファイル（`requirements.txt` 等）が正。ドキュメントには記載しない。

## 2. システムコンテキスト

### 2.1 外部インターフェース

<img src="diagrams/system-configuration.svg" alt="システム構成図" width="80%">

**図 2.1: システム構成図**

| 接続先 | プロトコル | 認証方式 | 使用ライブラリ方針 |
|---|---|---|---|
| 予約 API（reserve.central.co.jp） | HTTPS | Cookie（`device_id` + アクセストークン） | TLS 指紋互換の同期 HTTP クライアント（Chrome 相当） |
| 公開月間 API（www.central.co.jp） | HTTPS + JSONP | なし | 同上（JSONP のコールバックを剥がして JSON パース） |
| Discord Webhook | HTTPS | Webhook URL | 標準の同期 HTTP クライアントで十分 |
| 利用者ブラウザ | HTTPS | Basic 認証または Cookie | Web アプリフレームワーク標準 |
| シークレットストア（`/workspace/.secrets`） | ファイル読み取り | なし（ro マウント） | Fernet 復号（既存ユーティリティを再利用） |

予約 API への通信は市場-platform の同等処理と同じ方針で、Chrome の TLS 指紋と HTTP/2 プロファイルを再現するクライアントを使用する。これにより、受信側から見て通常のブラウザ操作と区別がつかない通信になる。

## 3. コード構造

### 3.1 コードマップ

`tools/central-sports-web/` 配下に次のディレクトリを配置する。

- `app/` — Web アプリ本体。プレゼンテーション層・アプリケーションサービス層・永続化層のモジュールをさらに分割する。
- `scheduler/` — プロセス内スケジューラのジョブ定義（月次同期、週次差分、日次準備、日次発火）。
- `client/` — 外部 API クライアント層。既存の事前調査用 CLI が使う `cs_api.py` と `cs_secrets.py` を共有 import する。
- `db/` — 永続化層。スキーマ定義と軽量マイグレーション、書き込みキュー。
- `ui/` — テンプレート・静的ファイル（CSS・必要最小限の JS）。
- `config/` — 設定ファイル（接続先・店舗デフォルト・通知設定等）。
- `docs/design/` — 本仕様書群と図。
- `docker-compose.yml` — コンテナ定義。

コードの詳細なファイル一覧はソース側が正とし、ドキュメントには書かない。

### 3.2 依存関係

- プレゼンテーション層 → アプリケーションサービス層 → （外部連携層 / 永続化層）。
- スケジューラ → アプリケーションサービス層（UI と同じユースケースを共用）。
- 外部連携層 → 外部サービス（双方向）。
- 永続化層 → 永続ストア（双方向、書き込みは直列化キューを介す）。
- 禁止: 外部連携層と永続化層が互いに直接依存すること。アプリケーションサービス層を経由させる。

### 3.3 拡張方法

新機能の追加は次のステップで行う:

1. アプリケーションサービス層にユースケース関数を追加する。
2. 必要に応じて外部連携層に API ラッパを追加する（既存 `cs_api.py` の拡張）。
3. 永続化層にテーブルや問い合わせを追加する。スキーマ変更時は軽量マイグレーションを記述する。
4. プレゼンテーション層（HTTP ルート + テンプレート）、もしくはスケジューラにエントリポイントを追加する。
5. 実行ログの記録点を追加する（構造化ログ + Discord 通知レベル）。

## 4. データ設計

### 4.1 ストレージ方式

組み込み RDB に WAL モードを有効化して使用する。同期書き込みはパフォーマンス上の要件を満たす範囲で NORMAL を選択する（クラッシュ耐性が必要なら FULL へ昇格する）。

アトミック書き込みはトランザクション単位で保証する。アプリケーションサービス層のユースケース 1 回に対してトランザクション 1 件を原則とし、スケジューラと UI の同時書き込みは単一の書き込みキューで直列化する。

### 4.2 状態管理

<img src="diagrams/target-state.svg" alt="ターゲット状態遷移図" width="70%">

**図 4.1: ターゲットの状態遷移**

主要なエンティティは次の 5 つ:

- **ターゲット**: 自動予約の定義。曜日・時刻・プログラム名パターン・希望席の配列・有効フラグ・状態（active / scheduled / running / succeeded / failed / skipped / disabled）。
- **予約**: 予約 API から返された結果のローカル記録。外部識別子・ローカル識別子・対象レッスン・席番号・状態。
- **実行ログ**: 外部 API 呼び出しの記録。リクエスト識別子・所要時間・エンドポイント・結果・マスク済みメタ情報。値はマスキング後のみ保存する。
- **スケジュールキャッシュ**: 月間テンプレートと直近実スケジュールのスナップショット。クエリキー（店舗・ルーム・年月・取得時刻）単位で保持し、TTL で再取得する。
- **店舗マスター**: 店舗・ルームの識別子と表示名。UI プルダウンの選択肢。

保持期間は実行ログが 90 日、予約結果が 365 日、スケジュールキャッシュが当月+前月（テンプレート）と直近 7 日（実スケジュール）を目安とする。

カラム定義・インデックス設計の詳細はスキーマ定義ファイルが正とし、ドキュメントには書かない。

### 4.3 バックアップ

週次ジョブで永続ストアの内容を JSON としてボリューム内にエクスポートし、コンテナ再作成に備える。復旧は JSON からの再 import で行う。シークレットはエクスポート対象外（メモリにしか存在しない）。

## 5. インフラ・デプロイ

### 5.1 環境構成

<img src="diagrams/deployment.svg" alt="配置図" width="70%">

**図 5.1: 配置図**

単一コンテナで全機能を稼働させる。ホストの Synology NAS 上に 2 種類のボリュームをマウントする。

- **ソース + 永続ストアボリューム**: アプリのソースコードと永続ストアを置く。読み書き可能。
- **シークレットボリューム**: マスター鍵と暗号化シークレットを置く。読み取り専用。

ポートは LAN 内のみ公開し、外部からの直接アクセスは許容しない。再起動方針は `restart: always` で NAS 再起動時にも自動復帰する。

### 5.2 パス設計

ホストの物理パスとコンテナ内パスは明確に分ける。コンテナ内ではアプリのソースが `/workspace/toolbox/tools/central-sports-web/` に配置される前提とし、シークレットは `/workspace/.secrets/` にマウントする（既存のシークレット連携方針に準拠）。

### 5.3 環境分離

運用は本番のみ。開発時はローカルマシンで `dry-run` フラグを有効にして予約 POST を抑止し、Discord 通知で内容確認のみ行う。テスト環境として別コンテナを立てる運用は本フェーズでは採用しない（単一プロセスの複雑度が低いため）。

## 6. 横断的関心事

### 6.1 セキュリティ実装

- **シークレットの復号**: 事前調査用の CLI と同じ復号ユーティリティを使用する。プロセス起動時に 1 回復号し、メモリ上で保持する。ファイルに書き戻さない。
- **マスキング境界**: 構造化ログ・Discord 通知・UI 応答の各出力境界で、メールアドレス・パスワード・アクセストークン・Cookie 値を `***` に置換する実装を 1 箇所にまとめる（各出力ポイントが自前で対応しない）。
- **Web UI 認証**: Basic 認証または Cookie ベース認証を必須とする。未認証リクエストはルーティング層で即座に弾く。
- **コンテナの egress 制限**: 外向き通信先を予約 API・公開月間 API・Discord Webhook の 3 点のみに限定する方針とする（具体的な iptables 等の設定は運用マニュアル側）。
- **通信互換性**: 予約 API への通信は TLS 指紋互換の HTTP クライアントで行う。ヘッダセット（UA、sec-ch-ua、sec-fetch-*、Accept 系）を Chrome 相当で統一する。

### 6.2 エラーハンドリング

予約 API の応答コードを次のカテゴリに分類し、振る舞いを決める。

| カテゴリ | 例 | 振る舞い |
|---|---|---|
| 認証失効 | トークン無効 | 1 回だけ再ログインを試行して retry |
| 検証エラー | 入力不正 | 実装バグとして扱い通知、再試行しない |
| ビジネスエラー | 満席・既予約・時間外 | 次の候補席を試行、もしくは停止 |
| 一時的サーバエラー | 5xx | 短時間バックオフで少数回 retry、超過で通知 |
| ネットワークエラー | timeout・DNS | 上記 5xx と同等扱い |

予約発火の失敗で通知しない分類は存在しない（何らかの応答は必ず行う）。

### 6.3 ログ

構造化ログ（JSON Lines）で出力する。各エントリに以下を含める: タイムスタンプ・ログレベル・リクエスト識別子・アプリケーションサービス関数名・所要時間・結果・マスク済みメタ情報。

Discord Webhook 通知は Embed 形式で、レベルに応じた色分けと件名・詳細を付ける。通知失敗は予約失敗とは切り分け、実行ログに記録のみ行う。

### 6.4 テスト戦略

- **単体テスト**: マスキング関数、ターゲットマッチングロジック、状態遷移、エラー分類判定。外部通信を伴わない純粋な関数を重点的に覆う。
- **統合テスト**: 予約 API への通信はモック（録画再生）で代替する。実サーバに対するテストはごく少数の smoke test のみ行う（本番アカウントへの副作用を避けるため dry-run 前提）。
- **スケジューラテスト**: 時刻固定のテストダブルで 9 時発火フローを通す。
- **カバレッジ**: 重要なユースケース（発火・エラー分類・状態遷移）を優先し、カバレッジ率は数値目標よりも回帰防止に注力する。

## 7. 技術的制約

### 7.1 環境制約

- Synology NAS 上の Docker で稼働。ホスト OS は Linux のため、Docker の `--no-sandbox` 系 Chromium を使う場合は注意。
- NAS の時刻同期（NTP）はホスト側で実施済みの前提。アプリ側は NTP 連携を行わない。
- 永続ストアのファイルはホストのボリューム上に置くため、btrfs スナップショットとの相互作用に注意する（ファイル監視系のメモリ消費の観点で、本件は永続ストアのみで影響は限定的）。

### 7.2 パフォーマンス制約

- 予約 API の POST は実測で 200 ms 前後。発火時刻を 9:00:00 JST に合わせた上で、ミリ秒単位の送信精度を保つため、スケジューラの `misfire_grace_time` と、Python 側の短時間 sleep による補正を組み合わせる。
- 9:00 直前に実スケジュール取得を行う関係で、準備ジョブ（8:55）と発火ジョブ（9:00）の間に外部通信が失敗する可能性があるため、直前の schedule 取得は複数段で段階的に行う（8:55 取得 → 9:00 直前に再確認）。
- 単一コンテナ・単一プロセスの前提で、スケジューラの同時ジョブ数は 1 に制限する。

## 8. 設計判断記録

### 採用: 常駐プロセス方式

**決定**: Docker コンテナを常駐させ、プロセス内にスケジューラとセッションキーパーを持つ。

**理由**: 9 時ちょうどの発火応答性を確保するため。毎回コンテナを起動する方式では 14 秒のオーバーヘッドが乗り、数秒の競争で敗北する。

**トレードオフ**: コンテナが落ちると発火機会を逃すリスクがある。`restart: always` と週次のバックアップで補完する。

### 採用: TLS 指紋互換の同期 HTTP クライアント

**決定**: 予約 API への通信は Chrome の TLS 指紋を再現する同期 HTTP クライアントを使用する。

**理由**: 標準の非同期 HTTP クライアントはデフォルトの TLS 指紋が独特で、通常のブラウザ通信と区別されやすい。市場-platform の既存 collector と同じ方針。

**トレードオフ**: 同期ライブラリのため非同期ループを阻害する。FastAPI のバックグラウンドスレッド機構で逃がすことで整合を取る。

### 採用: 組み込み RDB（SQLite 想定）

**決定**: 永続ストアに組み込み型の軽量 RDB を使う。

**理由**: 単一プロセス・低同時接続で運用コストが最小。バックアップも単一ファイルの export/import で完結する。

**トレードオフ**: 将来マルチプロセス化する際は別 RDB への移行が必要。本フェーズではその可能性は低い。

### 不採用: Playwright によるブラウザ自動操作

**決定**: 採用しない。HTTP クライアントレベルで十分。

**理由**: 起動コスト・メモリ消費が大きく、9 時発火の応答性に悪影響。TLS 指紋互換の HTTP クライアントで必要な互換性は得られる。

### 不採用: cron + シェルスクリプト

**決定**: 採用しない。

**理由**: 事前ログイン状態を保持できず、9 時発火の直前に毎回ログインが必要になる。ログイン所要時間（実測 487 ms）が読めないため、時刻精度を担保できない。
